# Technical Analysis & Implementation Design - vagas_bot Autonomous Recruitment

## Executive Summary
This report analyzes the existing `vagas_bot` codebase and details the architectural design needed to implement the 4 key pillars: (1) S-Tier Scrapers (LinkedIn, Glassdoor, InfoJobs), (2) Deep Scrape Snippet Bypass (Indeed, Jooble), (3) Groq AI Salary/Benefit Ranking, and (4) Auto-Apply Engine. Significant findings include database schema limitations (AI features are not stored in SQLite) and web app inconsistencies (FastAPI trigger bypasses the AI filter entirely).

---

## 1. Current Codebase Audit

### 1.1 Database Structure (`database.py`)
- **Schema Analysis**: The database `jobs.db` has a single table `jobs` initialized in `database.py`. 
- **Columns**: 
  - `id` (TEXT, Primary Key) - populated with the job `link`.
  - `title`, `company`, `budget`, `link`, `platform`, `added_at` (TIMESTAMP).
  - `job_type`, `profession`, `level`, `requirements` (added via `ALTER TABLE` try-except blocks).
- **Critical Flaws Found**:
  - The evaluation fields generated by Groq (such as `ai_score`, `ai_reason`, `ai_reqs`, `ai_bonus`, `ai_benefits`, and `ai_model`) are **never stored** in the database. `insert_jobs` only records the raw properties, discarding AI metadata.
  - There is no status tracking for application (e.g., `apply_status` like "pending", "applied", "failed") which is necessary for the Auto-Apply engine.

### 1.2 Scrapers Assessment (`scrapers/`)
- **LinkedIn (`linkedin.py`)**: Uses the anonymous Guest Search API (`/jobs-guest/jobs/api/seeMoreJobPostings/search`). It fetches only 50 jobs (pages 0 and 25) and fills the `requirements` field with a hardcoded placeholder containing the location. It **does not retrieve** the actual job description text.
- **Indeed (`indeed.py`)**: Uses Playwright to load the search page and extracts metadata from the `mosaic-provider-jobcards` JSON in the page content. It retrieves only a small `snippet` text, which is heavily truncated.
- **Jooble (`jooble.py`)**: Calls a POST endpoint on Jooble's API. It extracts a short `snippet` of the description, slicing it to 150 characters.
- **AI Filter (`ai_filter.py`)**: Configures Groq with 3 random API keys and a semaphore of 8 concurrency. Uses `llama-3.1-8b-instant`.
  - Filters out jobs based on rigid rules (e.g., wrong role, wrong location, senior level for juniors, dollar salaries for Brazilians, freelance tags).
  - Uses Pydantic `JobEvaluation` model to validate JSON responses from Groq.

### 1.3 Bot & Web App Orchestration (`bot.py`, `app.py`)
- **Telegram Bot (`bot.py`)**: 
  - Orchestrates hunts asynchronously using `asyncio.to_thread` for scraper modules.
  - Implements a pre-filter using `langdetect` to purge non-Portuguese listings and excludes freelance-related terms.
  - Performs resume-to-job matchmaking using Groq, and pushes the Top 5 matched jobs to the user.
- **FastAPI Web App (`app.py`)**:
  - Mounts a static web dashboard (`static/index.html`) using raw HTML/CSS/JS.
  - Exposes an `/api/trigger` endpoint which is invoked when the "Hunt Now" button is clicked.
  - **Inconsistency Warning**: `/api/trigger` in `app.py` runs scrapers and immediately inserts them into the DB **without running the Groq AI filter**. This leads to raw, unfiltered, and potentially foreign-language jobs entering the dashboard.

## 2. Pillar 1: S-Tier Scrapers Design

To satisfy the requirements of S-Tier scrapers fetching complete vacancy details:

```
               [ Scraper Search Loop ]
                          |
             Extracts Job ID / URL from cards
                          |
             [ Detailed Scraping Stage ]
             /            |            \
      (LinkedIn)     (Glassdoor)    (InfoJobs)
          |               |              |
    API Guest Detail  Playwright Stealth  curl_cffi Impersonate
          |               |              |
    Filter Employment   Full HTML DOM   Full description DOM
    Type == Full-time    extraction        extraction
```

### 2.1 LinkedIn S-Tier Scraper (`scrapers/linkedin.py` Rewrite)
- **Search Method**: Continue utilizing the Guest Search API to discover listings.
- **Deep Scrape Method**:
  1. Parse the job `link` from the search result. Extract the `job_id` using regex (`/view/(\d+)` or `currentJobId=(\d+)`).
  2. Request the anonymous mobile API endpoint: `https://www.linkedin.com/jobs-guest/jobs/api/jobPosting/{job_id}`.
  3. Parse the response HTML using BeautifulSoup.
  4. Extract the description from the `.show-more-less-html__markup` container.
  5. Check employment type: Look for the tag containing employment criteria and verify it contains "Tempo integral" or "Full-time". Discard the vacancy if it does not.
- **Purity Check (Unit Test)**:
  - Create a test verifying that the scraper populates the `requirements` field with a string of length > 500 characters and returns at least 10 valid full-time jobs.

### 2.2 Glassdoor Scraper (`scrapers/glassdoor.py`)
- **Challenge**: Glassdoor has aggressive Cloudflare security.
- **Implementation**:
  - Implement a Playwright scraper using stealth arguments (disable blink features, match viewport sizes, configure valid user-agent).
  - Target URL: `https://www.glassdoor.com.br/Vagas/index.htm?k={keyword}`.
  - Process:
    1. Open page in headless mode, wait for selector `.JobsList_jobSearchResult__y2Zca`.
    2. Extract individual job details links.
    3. Click on each card or open job link in a new tab.
    4. Extract full text from container `#JobDescriptionContainer`.

### 2.3 InfoJobs Scraper (`scrapers/infojobs.py`)
- **Implementation**:
  - Target URL: `https://www.infojobs.com.br/vagas-de-{keyword}.aspx`.
  - Use `curl_cffi` (impersonate="chrome110") to bypass TLS fingerprinting blocks.
  - Extract links of the form `https://www.infojobs.com.br/vaga-de-...___{job_id}.aspx`.
  - Fetch detail pages and parse the description from `.js_lblVagaDescricao` or container `#divDescricaoVaga`.

## 3. Pillar 2: Deep Scrape Snippet Bypass

For platforms sending only brief snippets (Indeed and Jooble), a secondary parsing step is required to extract full detail text before database insertion.

### 3.1 Indeed Deep Scraper (`scrapers/indeed.py`)
- **Optimization Strategy**: Loading a full page for every job on Indeed is resource-heavy and slow. To keep Playwright lightweight:
  1. Intercept network routes to block images, fonts, stylesheets, and analytics requests.
  2. For each job key extracted from the search results, open the job detail page: `https://br.indeed.com/viewjob?jk={jobkey}`.
  3. Wait for the selector `#jobDescriptionText` to load in the DOM.
  4. Read the text content of `#jobDescriptionText`.
  5. Replace the raw `snippet` in the job structure with the extracted text.

### 3.2 Jooble Deep Scraper (`scrapers/jooble.py`)
- **Strategy**: Jooble's API provides a redirect link.
  1. Follow the redirect link. If the destination is a Jooble host (`br.jooble.org/desc/...`), perform a direct HTTP GET request using `curl_cffi` (mimicking Chrome) or a Playwright page.
  2. Parse the HTML for the primary description container (typically annotated with `itemprop="description"` or class `_3e618`).
  3. Extract the full text, clean HTML markup, and update the job dict.

## 5. Pillar 4: Auto-Apply Engine

```
       [ Read CV: curriculo.txt ]
                  |
        [ Extract Profile Data ] (Name, Email, Tel, Links)
                  |
        [ Check Pending Jobs in DB ]
                  |
        [ Navigate to Apply URL ]
                  |
        [ Detect Input Fields ] (Playwright)
        /         |         \
   Text Input   File Upload  Checkboxes / Dropdowns
  (Name/Email)    (CV PDF)   (Agreement tags)
                  |
         [ Submit Form (Click) ]
                  |
     [ Verify HTTP 200 / DOM Alert ]
                  |
         [ Update Job DB Status ]
```

### 5.1 Architecture of `auto_apply.py`
- Read Diego's details from `curriculo.txt` (Name: Diego Antônio de Jesus Santos, Email: 9919622diego@gmail.com, Location: Assaí/PR).
- Establish a database utility that queries for jobs where `apply_status = 'pending'` and `ai_score >= 80`.
- Load the target application page using Playwright.
- Analyze the DOM to locate typical ATS form fields:
  - **Selectors**:
    - Name: `input[name*="name" i]`, `input[id*="nome" i]`
    - Email: `input[type="email" i]`, `input[name*="email" i]`
    - Phone: `input[type="tel" i]`, `input[name*="phone" i]`, `input[name*="celular" i]`
    - File upload: `input[type="file" i]` (interact with this to upload `temp_curriculo.pdf`)
  - **Action Flow**:
    1. Fill form fields sequentially.
    2. Upload `temp_curriculo.pdf`.
    3. Locate the primary submission button (e.g., `button[type="submit"]`, `input[type="submit"]`, or buttons containing words like "Candidatar", "Enviar", "Apply").
    4. Click the submission button and wait for confirmation.
